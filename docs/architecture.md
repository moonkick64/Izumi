# Izumi — architecture.md

## システム構成

```
[ソースツリー]
     ↓
[静的解析エンジン]      ←─ copyright / SPDX / LICENSE ファイルの抽出
     ↓
[ライセンス分類]        ←─ CONFIRMED / INFERRED / UNKNOWN に仕分け
     ↓
  ┌──────────────────────────────────────────────┐
  │  LLM SCA（オプション・ユーザーが選択）          │
  │                                              │
  │  オプション1: ローカルLLM ─→ OSSを特定        │
  │  オプション2: ローカルLLM（要約）              │
  │              ↓ ユーザーレビュー・編集          │
  │              外部LLM ─→ OSSを特定             │
  │  オプション3: 外部LLM ─→ OSSを特定            │
  └──────────────────────────────────────────────┘
     ↓
[ヒント提示]            ←─ 「XXライブラリに似ている可能性」をヒントとして表示
     ↓
[ユーザーレビュー]      ←─ 検出結果の確認・ライセンス変更・除外・コメント入力
     ↓
[SBOM出力]             ←─ SPDX 2.3 / CycloneDX 1.5
```

---

## 機能仕様

### フェーズ1: 静的解析エンジン

パッケージマネージャに依存せず、ソースツリーを直接スキャンする。

**スキャン対象:**
- ソースファイルのヘッダコメント（copyright表記、SPDXタグ）
- `LICENSE`, `COPYING`, `NOTICE`, `README` 等のファイル
- ディレクトリ名のヒューリスティック（`third_party/`, `vendor/`, `extern/`, `external/` 等）

**ライセンス分類（3段階）:**

| 分類 | 意味 | 例 |
|------|------|----|
| CONFIRMED | ライセンス・copyrightが明確 | SPDXタグあり、LICENSEファイルあり |
| INFERRED | 状況証拠から推定できる | `third_party/zlib/` にzlibのcopyright表記あり |
| UNKNOWN | 判断できない | copyrightなし・ライセンス記載なし |

CONFIRMED と INFERRED はユーザー操作なしに自動通過。UNKNOWN が存在する場合はユーザーへ警告を表示する。

---

### フェーズ2: LLM SCA（オプション）

ユーザーが任意で有効化できる。UNKNOWN に分類されたコードに対して、LLMを使ってOSSの混入を調査する。

#### LLMオプション1: ローカルLLMに直接質問

- UNKNOWNファイルを関数単位に分割
- 関数のソースコードをそのままローカルLLM（Ollama）に送信
- 「何のOSSが含まれているか特定してください」という趣旨のプロンプトで問い合わせ
- 結果をヒントとしてユーザーに提示

#### LLMオプション2: ローカルLLMで要約 → 外部LLMで特定

- UNKNOWNファイルを関数単位に分割
- 全関数をローカルLLM（Ollama）で**一括要約**する
  - 例：「バイト列を受け取りCRC32チェックサムを計算して返す関数」
  - 目的：ソースコードを抽象化し、機密情報を除いた形で外部送信可能な状態にする
- ユーザーが各関数の要約を確認・編集できる
- 「外部LLMで解析」ボタンを押すと、要約を1関数ずつ外部LLMに送信してOSSを特定させる
  - **外部LLMへの送信前に確認ダイアログを表示**
  - 送信するのはソースコードではなく要約テキスト（機密情報保護）
- 結果をヒントとしてユーザーに提示

#### LLMオプション3: 外部LLMに直接質問

- UNKNOWNファイルを関数単位に分割
- 関数のソースコードをそのまま外部LLMに送信
- 「何のOSSが含まれているか特定してください」という趣旨のプロンプトで問い合わせ
- 結果をヒントとしてユーザーに提示

#### プロンプトと構造化レスポンス

OSS特定を行うプロンプト（オプション1・2ステップ2・3）では、LLMに**JSON形式**でのレスポンスを要求する。

```json
{
  "component": "zlib 1.2.11",
  "license": "Zlib",
  "hint": "CRC32の実装とテーブル構造がzlibのcrc32()と一致します。"
}
```

- `component`: ライブラリ名とバージョン
- `license`: SPDX識別子
- `hint`: 判断理由の説明（UIのヒント欄に表示する）

特定できない場合は `"NOASSERTION"` を使用させる。

```json
{"component": "NOASSERTION", "license": "NOASSERTION", "hint": "該当するOSSが見つかりませんでした。"}
```

プロンプトは日本語・英語の2種類を用意し、UI言語設定（`i18n/`）に応じて自動で切り替わる。ユーザーは設定画面でプロンプトを変更できる。

**パース処理（`llm/prompts.py: parse_oss_response()`）:**
- JSONおよびmarkdownコードフェンス（` ```json ``` `）に対応
- パース成功時：`hint` をヒント欄に表示し、`component` / `license` をマッチングUIに自動入力
- パース失敗時：関数リストに `✗` マーカーを表示し、ヒント欄にエラーメッセージ＋生レスポンスを表示。ユーザーは再解析できる

#### 解析の単位

- すべてのオプションで**関数単位**で解析する
- 「解析」ボタンを押すと抽出済みの**全関数を一括**で順番に処理する（単一関数モードなし）
- 関数一覧はファイルを複数選択した場合、全ファイルの関数をまとめて表示する

#### 結果の永続化

- 解析結果はツール自身のデータディレクトリ（`~/.izumi/results/<project_name>_<hash>/llm_results.json`）に**自動保存**する
  - スキャン対象のソースツリーは汚染しない
  - プロジェクトが異なれば別ディレクトリに保存される
- 関数1件ごとに即時書き込み（解析が途中で中断されても結果が残る）
- レビュー画面を開いた際に既存の結果ファイルがあれば**自動ロード**する
- UIから結果ファイルを**削除**できる
- 保存フォーマット（JSON）:

```json
{
  "source_root": "/absolute/path/to/project",
  "created_at": "2026-03-28T12:00:00Z",
  "updated_at": "2026-03-28T15:30:00Z",
  "results": [
    {
      "file": "src/foo.c",
      "function": "my_func",
      "start_line": 42,
      "option": 1,
      "hint": "possibly similar to zlib crc32()",
      "matched_component": "zlib",
      "matched_license": "MIT"
    }
  ]
}
```

#### 結果の提示方法

- 「このコードはzlibのcrc32()関数に似ている可能性があります」のように提示
- 確定情報ではなくヒントである旨を明示（LLMのハルシネーション対策）
- ユーザーはヒントをもとに自分で調査・判断する

---

### フェーズ3: マッチング決定とSBOM出力

LLMのヒントをもとに、ユーザーが各関数のコンポーネント名とライセンスを決定する。

**マッチング決定:**
- LLMのヒントを参照しながら、コンポーネント名（例: `zlib`）とライセンス（例: `MIT`）をユーザーが手入力する
- 決定した情報は結果ファイルに保存される（`matched_component`, `matched_license`）
- 関数リストに決定済み（✓）・ヒントあり（~）・未解析（空白）のステータスを表示

**SBOMへの反映:**
- マッチング決定済みの関数を含むコンポーネントは、ユーザーが決定した名前・ライセンスをそのまま使用する
- SBOMにはディレクトリパス等のプロジェクト固有情報は含めず、OSS コンポーネントとしての情報のみを記録する
- コンポーネント名はディレクトリ名ではなく、ユーザーが同定したOSS名（例: `zlib`）を使用する

**SBOM出力フォーマット:**
- SPDX 2.3
- CycloneDX 1.5

**各コンポーネントの記録項目:**
- コンポーネント名（ユーザーが決定したOSS名）
- バージョン（判明している場合）
- ライセンス（ユーザーが決定）
- copyright表記
- 分類（CONFIRMED / INFERRED / UNKNOWN）

## 画面構成

### 画面1: プロジェクト設定
- スキャン対象ディレクトリの選択
- ローカルLLMの設定（Ollamaエンドポイント、使用モデル）
- 外部LLMの設定（APIキー、使用モデル）
- LLMへのプロンプト設定（デフォルトから変更可能）

### 画面2: スキャン結果一覧（メイン画面）
- Visual Studio ライクなレイアウト
  - 左ペイン：ファイルツリー（CONFIRMED / INFERRED / UNKNOWN のバッジ表示）
  - 右ペイン：選択ファイルのソースコードビューアー
- CONFIRMED / INFERRED / UNKNOWN 別のタブ表示
- UNKNOWN の件数を強調表示・警告
- 各ファイルのライセンス情報・変更・除外操作

### 画面3: LLM SCAレビュー画面

- LLMオプション選択（オプション1〜3）
- 対象ファイル選択・関数一覧（✓ / ~ / 空白のステータス表示）
  - ✓：マッチング決定済み
  - ~：LLMヒントあり（オプション2では要約生成済みを含む）
  - 空白：未解析
- 「解析」ボタン（一括処理のみ）
- 関数ソースコードビューアー
- **オプション2専用：** ローカルLLMで一括要約 → 要約テキストの確認 → 「外部LLMで解析」ボタンで一括送信
- LLMヒント表示エリア（確定情報ではない旨を明示）
- マッチング決定UI：コンポーネント名・ライセンスをユーザーが手入力して「Match」ボタンで確定
  - 決定結果は `llm_results.json` の `matched_component` / `matched_license` に保存
- 「結果削除」ボタン
- 「SBOM出力」ボタン（マッチング決定済みの情報をSBOMに反映）

### 画面4: SBOM出力
- コンポーネント一覧の最終確認
- 出力フォーマット選択（SPDX / CycloneDX）
- 出力先ファイルの指定

## 技術スタック

| 項目 | 選定 | 理由 |
|------|------|------|
| 実装言語 | Python | 指定 |
| GUIフレームワーク | PySide6 | 機能豊富・Visual Studio ライクなUI実装が可能 |
| ローカルLLM | Ollama + LiteLLM | LiteLLM経由で統一インターフェース |
| 外部LLM | LiteLLM | 140以上のプロバイダーに対応（Claude, GPT-4, Gemini, DeepSeek 等） |
| ソースパーサ | libclang（フォールバック：正規表現） | 下記参照 |
| SBOM出力 | spdx-tools / cyclonedx-python | 既存ライブラリを活用 |
| 多言語対応 | 独自辞書方式（i18n/） | 外部ライブラリ不要・シンプルな実装 |

### LLMインターフェース

ローカル・外部LLM ともに LiteLLM で統一する。設定ファイルでモデル名とAPIキーを管理し、コードの変更なしにプロバイダーを切り替えられる。

```python
from litellm import completion

# 外部LLM（Claude）
response = completion(model="claude-sonnet-4-20250514", messages=[...])

# 外部LLM（Gemini）
response = completion(model="gemini/gemini-2.0-flash", messages=[...])

# 外部LLM（DeepSeek）
response = completion(model="deepseek/deepseek-chat", messages=[...])

# ローカルLLM（Ollama）
response = completion(
    model="ollama/codellama",
    messages=[...],
    api_base="http://localhost:11434"
)
```

### 多言語対応（i18n）

UI文字列は `i18n/` モジュールで一元管理する。

**設計方針:**
- `.py` ソースファイルには日本語を含めない。文字列はすべて辞書ファイルに集約する
- デフォルト言語は英語。設定画面で日本語に切り替え可能
- 言語設定は `~/.izumi/config.json` の `"language"` キーに保存し、再起動で反映

**ファイル構成:**

```
i18n/
  __init__.py   # t(key, **kwargs) 関数・set_language() / get_language()
  en.json       # 英語文字列辞書（デフォルト）
  ja.json       # 日本語文字列辞書
```

**使い方:**

```python
from i18n import t

label = t("scan_start_btn")               # "Start Scan" or "スキャン開始"
msg   = t("invalid_path_msg", path=p)     # フォーマット付き
```

辞書にキーが存在しない場合はキー文字列をそのまま返す（フォールバック）。

### C/C++ 関数抽出

libclang を Python バインディング（`clang` ライブラリ）経由で使用する。

**2段構えのフォールバック設計:**

```
libclangで解析を試みる
    ↓ 成功 → ASTベースで関数を正確に抽出
    ↓ 失敗 → 正規表現ベースの簡易パーサにフォールバック
```

組み込みコードではコンパイラ固有の拡張（`__attribute__`、`#pragma`、ターゲット依存の `#ifdef` 等）が多く、libclangが解析エラーを出すケースがある。精度が落ちても止まらないことを優先する。

**libclangで取得できる情報:**
- 関数名・引数・戻り値の型
- 関数本文のソースコード範囲（開始行・終了行）
- インクルード関係

## 将来的な拡張候補

- **CI/CDパイプラインへの組み込み（CLIインターフェースの追加）**
  - 静的解析（フェーズ1）はCLI化により完全自動化できる
  - LLM SCA（フェーズ2）はCIで実行可能だが、マッチング確定はユーザーが行う必要があるため、CIは「ヒントレポート生成」にとどめるのが現実的
  - 実装方針：`main.py` からGUIを切り離し、`cli.py` として静的解析・LLM問い合わせを実行してJSONレポートを出力する形が適切
- 過去の調査結果のキャッシュ（同じコードの再調査を省略）
- チーム共有機能（調査結果・判断のエクスポート）
