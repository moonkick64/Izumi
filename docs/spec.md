# SBOM支援ツール 仕様書 (draft v0.1)

## 1. ツール概要

### 目的
C/C++組み込みソフトウェアのソースツリーから、未知のOSS混入を検出し、SBOMの作成を支援するツール。

### 設計思想
- **ツールは判断しない。ヒントを出す。** 最終判断は常にユーザーに委ねる
- **確認作業を最小化する。** 判断が必要なものだけをユーザーに提示する
- **信頼できる情報と不確かな情報を明確に分ける。** 曖昧なものをUNKNOWNとして明示する
- **機密情報の外部漏洩を設計で防ぐ。** ユーザーが確認・編集してから外部に送信する

---

## 2. システム構成

```
[ソースツリー]
     ↓
[静的解析エンジン]  ←─ copyright/SPDX/LICENSEファイルの抽出
     ↓
[結果の分類]        ←─ CONFIRMED / INFERRED / UNKNOWN に仕分け
     ↓
[ローカルLLM]       ←─ UNKNOWNのファイル・関数を要約（Ollama）
     ↓
[ユーザーレビュー]  ←─ 要約を確認・編集（機密除去）
     ↓
[外部LLM]          ←─ 要約をもとに類似OSSを問い合わせ
     ↓
[結果提示]          ←─ 「XXライブラリに似ている可能性」をヒントとして表示
     ↓
[SBOM出力]         ←─ SPDX / CycloneDX 形式
```

---

## 3. 機能仕様

### 3.1 静的解析エンジン（フェーズ1）

パッケージマネージャに依存せず、ソースツリーを直接スキャンする。

**スキャン対象:**
- ソースファイルのヘッダコメント（copyright表記、SPDXタグ）
- `LICENSE`, `COPYING`, `NOTICE`, `README` 等のファイル
- ディレクトリ名のヒューリスティック（`third_party/`, `vendor/`, `extern/`, `external/` 等）

**出力（3段階分類）:**

| 分類 | 意味 | 例 |
|------|------|----|
| CONFIRMED | ライセンス・copyrightが明確 | SPDXタグあり、LICENSEファイルあり |
| INFERRED | 状況証拠から推定できる | `third_party/zlib/` にzlibのcopyright表記あり |
| UNKNOWN | 判断できない | copyrightなし・ライセンス記載なし |

**CONFIRMEDとINFERREDはユーザーの確認作業不要（自動通過）。UNKNOWNのみ次フェーズへ。**

### 3.2 LLMによるOSS混入検出（フェーズ2）

UNKNOWNと分類されたコードに対して実行する。

**ステップ1: ローカルLLMによる要約（Ollama）**
- UNKNOWNファイルを関数単位に分割
- 各関数をローカルLLMに入力し、処理内容を自然言語で要約する
- 目的：機密性の高いソースコードを抽象化し、外部送信可能な形にする
- 要約例：「バイト列を受け取りCRC32チェックサムを計算して返す関数」

**ステップ2: ユーザーによる要約のレビュー**
- ローカルLLMが生成した要約をGUIに一覧表示
- ユーザーは各要約を確認し、以下の操作が可能：
  - そのまま承認して外部LLMに送信
  - 要約を編集してから送信（機密情報が含まれる場合）
  - スキップ（外部LLMに送らない）
- **外部LLMへの送信はユーザーが明示的に承認したもののみ**

**ステップ3: 外部LLMへの問い合わせ**
- 承認された要約を外部LLM（Claude / GPT-4等）に送信
- プロンプト例：「以下の処理を行う関数に類似したオープンソースライブラリや関数を教えてください。もし心当たりがなければその旨を答えてください。」
- 外部LLMの回答をそのまま「調査ヒント」としてユーザーに提示

**ステップ4: ユーザーへのヒント提示**
- 「このコードはzlibのcrc32()関数に似ている可能性があります」
- 確定ではなくヒントである旨を明示（LLMのハルシネーション対策）
- ユーザーはヒントをもとに自分で調査・判断する

### 3.3 SBOM出力（フェーズ3）

ユーザーが各コンポーネントの情報を確認・確定した後、SBOMを出力する。

**出力フォーマット:**
- SPDX 2.3
- CycloneDX 1.5

**各コンポーネントの記録項目:**
- コンポーネント名
- バージョン（判明している場合）
- ライセンス
- copyright表記
- 分類（CONFIRMED / INFERRED / UNKNOWN）
- 調査ヒント（LLMの提示内容）
- ユーザーの判断・コメント

## 4. 画面構成（デスクトップGUIアプリ）

### 画面1: プロジェクト設定
- スキャン対象ディレクトリの選択
- ローカルLLMの設定（Ollamaのエンドポイント、使用モデル）
- 外部LLMの設定（APIキー、使用モデル）

### 画面2: スキャン結果一覧
- CONFIRMED / INFERRED / UNKNOWN 別のタブ表示
- ファイル・ディレクトリ単位でコンポーネントを一覧表示
- UNKNOWNの件数を強調表示

### 画面3: UNKNOWNレビュー画面
- 対象ファイル・関数の一覧
- ローカルLLMが生成した要約の表示・編集
- 外部LLMへの送信ボタン（個別 / 一括）
- 外部LLMからのヒント表示
- ユーザーのコメント入力欄

### 画面4: SBOM出力
- コンポーネント一覧の最終確認
- 出力フォーマット選択（SPDX / CycloneDX）
- 出力先ファイルの指定

## 5. 技術スタック

| 項目 | 選定 | 理由 |
|------|------|------|
| 実装言語 | Python | 指定 |
| GUIフレームワーク | PySide6 | 機能豊富・見た目が良い |
| ローカルLLM | Ollama + LiteLLM | LiteLLM経由で統一インターフェース |
| 外部LLM | LiteLLM | 140以上のプロバイダーに対応（Claude, GPT-4, Gemini, DeepSeek, Qwen等） |
| ソースパーサ | libclang（フォールバック：正規表現） | 下記参照 |
| SBOM出力 | spdx-tools / cyclonedx-python | 既存ライブラリを活用 |

### LLMインターフェースの設計

ローカル・外部LLMともにLiteLLMで統一する。ユーザーは設定画面でモデルを自由に切り替えられる。

```python
from litellm import completion

# 外部LLM（Claude）
response = completion(model="claude-sonnet-4-20250514", messages=[...])

# 外部LLM（Gemini）
response = completion(model="gemini/gemini-2.0-flash", messages=[...])

# 外部LLM（DeepSeek）
response = completion(model="deepseek/deepseek-chat", messages=[...])

# ローカルLLM（Ollama）
response = completion(
    model="ollama/codellama",
    messages=[...],
    api_base="http://localhost:11434"
)
```

設定ファイルでモデル名とAPIキーを管理し、コードの変更なしにプロバイダーを切り替えられる。

C/C++の関数抽出にはlibclangをPythonバインディング（`clang`ライブラリ）経由で使用する。

**2段構えのフォールバック設計:**

```
libclangで解析を試みる
    ↓ 成功 → ASTベースで関数を正確に抽出
    ↓ 失敗 → 正規表現ベースの簡易パーサにフォールバック
```

組み込みコードではコンパイラ固有の拡張（`__attribute__`、`#pragma`、ターゲット依存の`#ifdef`等）が多く、libclangが解析エラーを出すケースがある。その場合は精度が落ちても止まらないことを優先する。

**libclangで取得できる情報:**
- 関数名・引数・戻り値の型
- 関数本文のソースコード範囲（開始行・終了行）
- インクルード関係

## 6. スコープ外（意図的に含めない機能）

- スニペットマッチング（DBとのハッシュ照合）
- 脆弱性スキャン（CVE検出等）
- クラウド連携・SaaS提供
- パッケージマネージャ対応（npm, pip等）
